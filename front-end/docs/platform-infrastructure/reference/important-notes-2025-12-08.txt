Phase 1 – Backend: multi-bucket OCR uploads
--------------------------------------------
1) Locate the OCR upload route and job model:
   - Find the router/controller handling:
     - `POST /api/church/:churchId/ocr/upload`
   - Find the DB access code and type definitions for `ocr_jobs` (or equivalent), e.g.:
     - `ocrJobsRepo`, `OcrJob`, etc.

2) Add support for **upload buckets**:
   - Concept: a "bucket" is a small enum determining *where* on disk the image is stored and how we categorize the job.
   - Buckets:
       - `records`   → production record scans (current default)
       - `test`      → ad-hoc test uploads
       - `benchmark` → curated benchmark datasets for accuracy testing

3) Environment/config:
   - Add these env/config values (if not present) to the server:
     - `OCR_UPLOAD_ROOT=/var/www/orthodoxmetrics/ocr_uploads`
     - `OCR_BUCKETS=records,test,benchmark`
   - Implement a helper:
     - `resolveBucket(req: Request): 'records' | 'test' | 'benchmark'`
       - Check req.query.bucket and req.body.bucket.
       - If value is not in `OCR_BUCKETS`, default to `records`.

4) Update multer storage for OCR uploads:
   - Where we currently decide the `destination` for uploaded images, replace the fixed path with:
     - `const bucket = resolveBucket(req);`
     - Target directory: `${OCR_UPLOAD_ROOT}/${bucket}`
     - `fs.mkdirSync(dir, { recursive: true })` before writing.
   - Keep filename logic similar (timestamp + random + extension).

5) Extend `ocr_jobs` schema:
   - Add a `bucket` column (if it does not exist):
     - `VARCHAR(32) NOT NULL DEFAULT 'records'`.
   - Update any insert logic to set the job’s `bucket` based on `resolveBucket(req)` and the chosen upload directory.
   - Make sure `OcrJob` types/interfaces include `bucket`.

6) Add optional bucket filtering to job listing:
   - For `GET /api/church/:churchId/ocr/jobs`, accept optional query param `bucket`.
   - If provided and valid, filter jobs by that bucket.
   - If omitted, default to “all buckets” (for now).
   - Keep this change backwards-compatible: existing callers with no `bucket` work as before.

7) Ensure the OCR processing worker (if separate) can handle the updated `ocr_jobs` rows and does not break when `bucket` is present.

Phase 3 – Batch OCR benchmark runner (backend script)
------------------------------------------------------
Goal: be able to point at a folder of images (and optional ground-truth text) and compute OCR accuracy metrics.

1) Create a new Node script in the server side, something like:
   - `server/scripts/runOcrBenchmark.ts` (or `.js` if the server is JS-only).

2) Script behavior:
   - Usage:
     - `node dist/scripts/runOcrBenchmark.js /path/to/folder --churchId=<id> --bucket=benchmark`
   - Inputs:
     - Required: folder path.
     - Optional: churchId (for calling church-specific OCR endpoint) and bucket (for logical grouping).
   - For each image file in the folder (`.jpg`, `.jpeg`, `.png`, `.tif`, `.tiff`):
     - Call the existing OCR stack, using the internal logic or HTTP endpoint:
       - Preferred: call the internal OCR function/service (no HTTP) if that’s easily importable.
       - Otherwise, send HTTP POST to:
         - `/api/test-ocr` OR `/api/church/:churchId/ocr/upload` + job polling, whichever is cleaner.
     - Look for ground-truth:
       - For `img001.jpg`, check alongside:
         - `img001.txt` (plain expected text), or
         - `img001.json` (structured expected fields).
       - If none exist, still store the OCR output but mark accuracy as `null`.

3) Metrics:
   - Implement basic text accuracy metrics:
     - Character-level Levenshtein distance → character accuracy %.
     - Word-level overlap / word accuracy % (simple tokenization on spaces/punctuation is fine).
   - For JSON ground truth (optional), compute simple field accuracy:
     - For each key, compare expected vs actual string values.
     - Record how many fields match exactly and field accuracy %.

4) Output:
   - Write:
     - `benchmark_results.json` in the folder.
     - `benchmark_results.html` with:
       - summary (num files, avg char accuracy, avg word accuracy, number with ground truth).
       - table per image:
         - filename
         - OCR status (ok/error)
         - char accuracy %
         - word accuracy %
         - link or path to raw OCR output
         - note if ground truth is missing.

   - Keep the HTML layout simple but readable (similar style to the Python `image_enhancer` quality report: table, basic CSS, summary at the top).

5) Implementation details:
   - Reuse existing HTTP client utilities or import the OCR logic directly where possible (do not rewrite OCR).
   - Add logging but no interactive prompts.
   - Exit with non-zero code if the script itself fails; individual image failures should be logged but not crash the entire run.
