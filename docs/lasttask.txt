Goal: Migrate internal dev/ops tools from OrthodoxMetrics into OMAI in a controlled sequence, making OMAI the operational control plane and avoiding duplicate ownership. Before making changes, identify and document the current setup, dependencies, backend routes, DB touchpoints, auth rules, and menu/router registrations for each tool. Then migrate them in phases, verify each cutover, and decommission the old OrthodoxMetrics versions after validation.

Migration order to implement

Phase 1 — pure control plane
1. Build Console
2. Platform Status
3. Conversation Log
4. API Explorer

Phase 2 — dev architecture tools
5. OMTrace Console
6. Router/Menu Studio
7. Menu Editor
8. Page Content Editor
9. Refactor Console

Phase 3 — hybrid orchestration/inspection tools
10. Dynamic Records Inspector
11. Interactive Report Jobs
12. OCR Operations
13. OCR Batch Manager
14. OM Permission Center

Core architectural rules
- OMAI should own internal ops/dev tooling
- OrthodoxMetrics should own product/runtime church behavior
- Hybrid tools should have UI/orchestration in OMAI but interact with OrthodoxMetrics through explicit connectors/APIs
- Do not keep duplicate long-term versions in both systems
- Move -> validate -> switch -> decommission

For each tool, do this process:
1. Current-state audit
- frontend components/pages
- backend routes
- DB access
- auth/role checks
- menu/router entries
- dependencies on OrthodoxMetrics-specific code
- exact files to modify

2. Ownership decision
Classify each tool as one of:
- fully OMAI-owned
- OMAI-owned orchestration UI with OrthodoxMetrics-connected backend/data
Document the reasoning.

3. Migration implementation
- move/adapt frontend into OMAI
- move/adapt backend into OMAI where appropriate
- for hybrid tools, replace direct embedded product logic with explicit API/service boundaries
- preserve permissions and operational behavior

4. Validation
- route loads
- menu entry works
- auth works
- backend actions work
- no broken dependencies
- no silent fallback to old OrthodoxMetrics paths

5. Decommission
After each tool is validated in OMAI:
- remove or disable OrthodoxMetrics menu entry
- remove or disable old OrthodoxMetrics route/page
- remove duplicate write paths
- keep overlap as short as possible

Required output before implementation
Produce:
- migration matrix by tool
- recommended ownership type
- dependency/risk list
- proposed batch plan
- exact files that will be touched

Required final output
For each migrated tool provide:
- files changed
- route changes
- menu changes
- backend/API changes
- whether it is fully OMAI-owned or OMAI-orchestrated
- what was decommissioned in OrthodoxMetrics
- verification results
